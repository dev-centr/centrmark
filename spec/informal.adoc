= CentrMark Informal Specification
:toc: macro
:sectnums:
:sectlinks:

This document specifies the informal semantics and syntax constraints of CentrMark (`.cmk`), contrasting it to its predecessors like Markdown, Asciidoc, and MDX.

toc::[]

== 1. The File Structure
Every CentrMark document evaluates as a sequence of blocks. Optionally, the document can start with a **Frontmatter** metadata block. Unlike Markdown which standardizes on YAML or JSON, CentrMark uses **SDL (Simple Declarative Language)**.

=== 1.1 Frontmatter
SDL relies on unquoted keys, implicit arrays, and block structures cleanly delimited without noisy colons or equal signs. It must be bounded by `---` at the beginning and end.

[source,centrmark]
----
---
title "CentrMark Informal Spec"
date 2026-03-13
author {
  name "Jane Writer"
  email "jane@dev-centr.com"
}
tags "spec", "draft", "technical"
---
----

== 2. Document Primitives (Markdown Continuity)
CentrMark preserves the most intuitive text features from CommonMark. If a document only contains standard markdown, it parses correctly under CentrMark.

* **Headers**: `# H1`, `## H2`, `### H3`
* **Emphasis**: `*strong*` / `**strong**`, `_emphasis_` / `__emphasis__`.
* **Inline Code**: Backticks (`` ` ``).
* **Code Blocks**: Triple backticks (`` ``` ``) with optional language identifiers.

=== 2.1 Fenced Blockquotes
Markdown's `> ` prefix for quotes is a known friction point that breaks copy-paste workflows (especially when pasting code inside a quote). CentrMark solves this elegantly with **Fenced Blockquotes** using `((` and `))`.

These parenthesis clusters visually cradle the text like a giant typographic pull-quote. Crucially, they do not conflict with string literals in programming languages, and because they are directional, they can be nested cleanly without requiring varying lengths.

[source,centrmark]
----
((
The limit of MDX is when technical writers have to learn React.
-- A wise writer
((
This is a nested quote!
))
))
----

=== 2.2 Advanced Lists
Lists in CentrMark support all markdown basics but resolve historical ambiguities regarding numbering, nesting, and continuation.

==== 2.2.1 Auto-Incrementing Ordered Lists
Adjusting numbers in a long list is tedious. While you can use `1.` repeatedly, CentrMark formally adopts `0.` as the generic, auto-incrementing ordered list marker.

Cognitively, `0` serves as a terminal exemplar (a real number) that signals an array or list starting point without requiring physical shifted keys (like `#`).

[source,centrmark]
----
0. Wash the car
0. Dry the car
0. Wax the car
----

==== 2.2.2 List Flattening and Nesting
You can freely mix bullets (`*`, `-`, `+`) and numbers (`0.`, `1.`). 

CentrMark supports two forms of nesting:
1. **Spatial Indentation**: Using the negative space (margin) on screen to denote hierarchy. It is naturally limited only by screen width.
2. **Symbol Alternation (Flattening)**: Explicitly dropping down a list level without changing indentation simply by alternating terminal symbols. Because you run out of standard symbols (`*`, `-`, `+`) quickly, extreme nesting levels are highly discouraged when flattening.

[source,centrmark]
----
* Outer Bullet
- Inner Bullet (Flattened Nesting)
0. Inner Numbered List item 1
0. Inner Numbered List item 2
* Back to outer bullet (Flattened return)
----

**Virtual Indentation Environments (LSP Feature)**:
Because reading deeply structured data in flattened space is cognitively difficult, the CentrMark specification strongly advises code editors and language servers (LSPs) to implement **Virtual Indentation environments**. Editors must interpret symbol changes (recursion levels) and visually shift the active text editing zone to the right while rendering a slightly different background color for the text layer versus the underlying editor window background. This creates visual scoping without writing whitespace.

==== 2.2.3 List Continuation
A major flaw in Markdown is attaching subsequent blocks (like code snippets or paragraphs) to a list item using fragile exact-space indentation. CentrMark adopts AsciiDoc's highly successful List Continuation marker: the `+` on a line by itself.

[source,centrmark]
----
0. Modify the configuration file:
+
```json5
{ "setting": true }
```
+
Ensure you save it before running the service.
0. Restart the service.
----


== 3. Extensibility Mechanisms
Where Markdown fails is its lack of standard structural markup (admonitions, tabs, figures). CentrMark replaces all custom flavor ad-hoc parsers with three uniform **Directive** mechanisms: Inline, Void, and Block.

=== 3.1 Properties and Typings
Directives optionally take strongly typed properties declared via an inline fragment of **SDL** wrapped in square brackets `[ ... ]`.

=== 3.2 Inline Directives
Used within spans of text. An inline directive begins with an `@` character followed by its name. Like macros in Confluence, they enable rich integrations.
Syntax: `@name[properties](content)`

Examples:
* `@badge[color="red"](New)` -> Renders a New red badge.
* `@mention[user="rjamd"]` -> Renders a profile link to user 'rjamd'.

=== 3.3 Void Directives
Used to render block-level elements that do not contain markdown content. They start with two colons `::` at the beginning of a line.

Syntax: `:: name [properties]`

Examples:
* `:: toc [depth=3]`
* `:: include [file="./partials/footer.cmk" vars={version: "1.2"}]`

=== 3.4 Block Directives
Used to construct nested structural elements containing markdown or other directives (Accordions, Tabs, Complex Admonitions, Code Groups). They start with a sequence of colons (fence), minimum of three `:::`. The block is terminated by a matching sequence of colons.

Syntax:
[source,centrmark]
----
::: name [properties]
Markdown content or sub-blocks...
:::
----

Example: **Tabs**
MDX forces `<Tabs><TabItem value="apple">...</TabItem></Tabs>`. AsciiDoc forces noisy custom block macros. CentrMark is contextually clean:

[source,centrmark]
----
:::: tabs [sync="os-type"]
::: tab [label="Windows"]
```cmd
echo "Running on Windows"
```
:::

::: tab [label="Linux"]
```bash
echo "Running on Linux"
```
:::
::::
----

=== 3.5 Admonitions
A special subset of Block Directives heavily used in technical writing.
[source,centrmark]
----
::: note [title="Performance Warning" icon="warning"]
Be careful configuring this setting, it can exponentially increase latency.
:::
----

=== 3.6 Typed Checklists
Checklists are Block Directives whose body is parsed as bracket-marked task lines.

Syntax:
[source,centrmark]
----
::: checklist [type="default"]
[ ] Unchecked
[x] Done
[-] Blocked or pending
:::
----

Default markers:

* `[ ]` unchecked
* `[x]` complete
* `[-]` blocked/pending

`type` allows renderers to apply alternate semantics and visuals while preserving the same bracket-marker syntax. Recommended built-in types are `default`, `progress`, `decision`, and `risk`.

=== 3.7 Diagram Blocks
Diagrams are represented with a `diagram` Block Directive. The body contains source text (for example Mermaid, PlantUML, or Graphviz DOT), or `src` can reference a pre-rendered asset.

[source,centrmark]
----
::: diagram [format="mermaid"]
flowchart TD
  A[Start] --> B{Ready?}
  B -->|yes| C[Ship]
  B -->|no| D[Revise]
:::
----

Supported-by-default diagram formats in the reference renderer:

* `mermaid`
* `plantuml`
* `graphviz` / `dot`

=== 3.8 Animation Blocks
Animation assets are represented with an `animation` Block Directive and format metadata.

[source,centrmark]
----
::: animation [format="lottie" src="/animations/intro.json" autoplay=true loop=true]
:::
----

`lottie` is the recommended default format for structured, scriptable animation payloads. Renderers may also support media formats such as `gif`, `mp4`, or `webm`.

== 4. Advanced Markdown Enhancements

=== 4.1 Definition Lists
A common omission from basic Markdown. CentrMark utilizes the tilde `~` symbol at the beginning of a line for the term, and indented content below it.
[source,centrmark]
----
~ API Route
  An endpoint that responds with JSON data.
~ Static Page
  A pre-compiled HTML page served rapidly from CDN.
----

=== 4.2 Structural Tables
Basic markdown tables are supported (`| Header |`). For advanced tables (spanning columns, rich markdown cells), CentrMark uses Block Directives.
[source,centrmark]
----
::: table [columns="1fr 2fr"]
:: row
  ::: col [span=2 align="center"]
  **Header spanning both columns**
  :::
:: row
  ::: col
  Cell 1 (Supports multiple lines, nested lists, images)
  :::
  ::: col
  Cell 2
  :::
:::
----

=== 4.3 Semantic Cross-Links
Standard markdown links `[text](url)` work. Semantic cross links are bracketed by `[[` and `]]`.
* `[[#setup | Read the setup guide]]`
* `[[user-guide#login]]`

== Summary of Keyboard Economy
The CentrMark standard entirely restricts itself to commonly accessible, low-friction keyboard patterns:
* `.` `,` `/` `[` `]` `{` `}` `<` `>` `(` `)`: Present, standard.
* `:` (Colon): Dominantly used for blocks, physically easier to type than repetitive backslashes or equals signs.
* `@` (At-sign): Ubiquitous for internet phenomena, extremely quick to parse visually.
* `-` `*` `_` `#` and `~`: All retained from successful markdowns. 
* `+`: Simple bridging cross for list continuations.

Absolutely zero raw HTML/XML tags (`<tag>`) are required, dramatically cleaning raw text.
